# Support

## Community Support

EdgeShield has a growing community of users and contributors. The following resources are available for community support:

### GitHub Discussions

Use [GitHub Discussions](https://github.com/edgeshield/edgeshield/discussions) for:

- Questions about installation, configuration, and usage
- Show and tell (share your EdgeShield setup)
- Ideas and feature requests
- General discussion

### Discord

Join the [EdgeShield Discord server](https://discord.gg/edgeshield) for real-time community support:

- `#help` — Get help with installation and configuration
- `#showcase` — Share your EdgeShield deployments
- `#development` — Discuss development and contribute
- `#announcements` — Release announcements and updates

### GitHub Issues

Use [GitHub Issues](https://github.com/edgeshield/edgeshield/issues) for:

- Bug reports (see [Issue Reporting](#issue-reporting) below)
- Feature requests (see [Feature Requests](#feature-requests) below)

## Issue Reporting

### Before reporting

1. **Search existing issues**: Check if the issue has already been reported.
2. **Check the FAQ**: See [README.md](README.md#faq) for common questions.
3. **Check the documentation**: Ensure you are using the correct configuration.
4. **Update to the latest version**: The issue may have been fixed in a newer release.

### Reporting a bug

Open a [GitHub Issue](https://github.com/edgeshield/edgeshield/issues/new) with the following information:

- **EdgeShield version**: Output of `edgeshield --version`
- **System information**: OS, architecture, kernel version
- **Hardware**: Raspberry Pi model or other hardware
- **Configuration**: Your config file (redact sensitive information)
- **Steps to reproduce**: Minimal, complete, and verifiable steps
- **Expected behavior**: What you expected to happen
- **Actual behavior**: What actually happened
- **Logs**: Relevant log output (use `RUST_LOG=debug` for detailed logs)
- **Screenshots**: If applicable

### Security issues

**Do not** file a public GitHub issue for security vulnerabilities. See [SECURITY.md](SECURITY.md) for the security vulnerability reporting process.

## Feature Requests

Feature requests are welcome. Open a [GitHub Issue](https://github.com/edgeshield/edgeshield/issues/new) with the following:

- **Clear description**: What the feature should do
- **Use case**: Why this feature is useful and who it benefits
- **Expected behavior**: How the feature should work
- **Alternatives**: Any workarounds or alternative solutions you've considered
- **Priority**: How important this feature is to you (nice-to-have, important, critical)

Feature requests are discussed and prioritized by the maintainers. Community feedback on feature requests helps us prioritize.

### Feature request lifecycle

1. **Submitted**: Feature request is opened as a GitHub issue
2. **Discussed**: Community and maintainers discuss the proposal
3. **Triaged**: Maintainers assign priority and milestone
4. **Designed**: If accepted, a design document may be created
5. **Implemented**: Feature is implemented in a pull request
6. **Released**: Feature is included in a release
